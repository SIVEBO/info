SIVEBO — Guia completa

Java, Spring Boot, Microservicios, Docker y todo lo que necesitas para dominar y defender tu proyecto.

Que es Java

Java es un lenguaje de programacion orientado a objetos, compilado y tipado estaticamente. Tu codigo .java se compila a bytecode (.class) que corre en la JVM (Java Virtual Machine), lo que lo hace multiplataforma.

// Todo programa Java vive dentro de una clase
public class HolaMundo {
    public static void main(String[] args) {
        System.out.println("Hola mundo");
    }
}
Dato clave: En SIVEBO usamos Java 25 con Spring Boot 4.0.6. Spring Boot elimina toda la configuracion manual que Java EE necesitaba.

Tipos de datos

Tipos de referencia importantes:

String nombre = "SIVEBO";                 // Cadena de texto
BigDecimal precio = new BigDecimal("1500.00"); // Para dinero (preciso)
LocalDate hoy = LocalDate.now();           // Fecha sin hora

Nunca uses double para dinero. Usa BigDecimal. Los decimales flotantes pierden precision: 0.1 + 0.2 = 0.30000000000000004. En SIVEBO todos los montos usan BigDecimal.

POO: Clases y objetos

Todo en Java es una clase. Una clase es un plano; un objeto es una instancia de ese plano.

public class CategoriaEmbalaje {

    // Atributos (estado)
    private Long idCat;
    private String nombreCategoria;

    // Constructor
    public CategoriaEmbalaje(Long idCat, String nombreCategoria) {
        this.idCat = idCat;
        this.nombreCategoria = nombreCategoria;
    }

    // Getters y Setters (acceso controlado)
    public Long getIdCat() { return idCat; }
    public void setIdCat(Long idCat) { this.idCat = idCat; }

    public String getNombreCategoria() { return nombreCategoria; }
    public void setNombreCategoria(String n) { this.nombreCategoria = n; }
}

Los 4 pilares de la POO:

  1. Encapsulamiento — atributos private, acceso via getters/setters
  2. Herencia — una clase hija hereda de una padre con extends
  3. Polimorfismo — un mismo metodo se comporta diferente segun la clase
  4. Abstraccion — interfaces y clases abstractas definen el "que" sin el "como"

Herencia e interfaces

// INTERFAZ: define un contrato (el "que")
public interface ArticuloEmbalajeService {
    ArticuloEmbalajeResponse crear(ArticuloEmbalajeRequest request);
    ArticuloEmbalajeResponse obtenerPorId(Long id);
    List<ArticuloEmbalajeResponse> listarTodos();
}

// IMPLEMENTACION: cumple el contrato (el "como")
public class ArticuloEmbalajeServiceImpl implements ArticuloEmbalajeService {

    @Override
    public ArticuloEmbalajeResponse crear(ArticuloEmbalajeRequest request) {
        // aqui va la logica real
    }
}
Por que usamos interfaz + impl en SIVEBO: El controller depende de la interfaz, no de la implementacion. Si manana quieres cambiar la logica, creas otra impl sin tocar el controller. Este es el Principio de Inversion de Dependencias (la D de SOLID).

Collections y Streams

Las estructuras de datos mas usadas:

List<String> nombres = new ArrayList<>(); // Lista ordenada
nombres.add("Caja");
nombres.add("Sobre");

Map<String, String> errores = new HashMap<>(); // Clave-valor
errores.put("nombre", "El nombre es obligatorio");

Streams — procesan colecciones de forma funcional. Los usamos mucho en SIVEBO:

// Convertir lista de entidades a lista de DTOs
List<ArticuloEmbalajeResponse> respuesta = repository.findAll()
    .stream()                          // abrir el stream
    .map(this::toResponse)              // transformar cada elemento
    .collect(Collectors.toList());      // recolectar en lista

// Filtrar: solo activos
List<Articulo> activos = articulos.stream()
    .filter(a -> a.getActivo())
    .collect(Collectors.toList());

Excepciones

Las excepciones manejan errores sin crashear la app:

// Excepcion personalizada (la creamos nosotros)
public class RecursoNoEncontradoException extends RuntimeException {
    public RecursoNoEncontradoException(String mensaje) {
        super(mensaje);
    }
}

// Como se usa:
ArticuloEmbalaje articulo = repository.findById(id)
    .orElseThrow(() -> new RecursoNoEncontradoException("Articulo no encontrado con id: " + id));

Checked vs Unchecked:

  • RuntimeException (unchecked) — no obliga a hacer try/catch. Nuestras excepciones heredan de esta.
  • Exception (checked) — obliga al llamador a manejarla. Ejemplo: IOException.

Optional

Evita NullPointerException. Envuelve un valor que puede o no existir:

// findById retorna Optional<T>
Optional<ArticuloEmbalaje> maybe = repository.findById(1L);

// Forma segura: lanza excepcion si no existe

    .orElseThrow(() -> new RecursoNoEncontradoException("No encontrado"));

// Otras formas:
maybe.isPresent();            // true si tiene valor
maybe.orElse(valorDefault);   // retorna default si vacio
maybe.ifPresent(a -> ...);    // ejecuta lambda si tiene valor

Que es Spring Boot

Spring Boot es un framework que simplifica la creacion de aplicaciones Java. Provee:

  • Auto-configuracion — detecta tus dependencias y configura todo solo
  • Servidor embebido — incluye Tomcat, no necesitas instalar nada aparte
  • Starter dependencies — paquetes predefinidos que agrupan todo lo que necesitas
  • Inyeccion de dependencias — Spring crea y conecta los objetos por ti
@SpringBootApplication  // Punto de entrada
@EnableDiscoveryClient // Se registra en Eureka
public class MsEmbalajeApplication {
    public static void main(String[] args) {
        SpringApplication.run(MsEmbalajeApplication.class, args);
    }
}
Inyeccion de dependencias explicada: Cuando escribes @RequiredArgsConstructor en un Service y declaras private final ArticuloEmbalajeRepository repository;, Spring automaticamente busca una implementacion de ese tipo, la crea, y la inyecta en tu clase. Tu nunca haces new ArticuloEmbalajeRepository().

Arquitectura en capas

Cada microservicio de SIVEBO sigue esta arquitectura:

Cliente (HTML / Postman / otro MS) | v Controller ← Recibe HTTP, valida, delega | v Service ← Logica de negocio | v Repository ← Acceso a base de datos (JPA) | v Base de datos (MariaDB)

Ademas, usamos DTOs para separar lo que entra/sale de la API vs lo que se guarda en BD:

Request DTO → Controller → Service → Entity → BD ↓ Response DTO ← Controller ← Service ← Entity ← BD

Entity (Modelo)

Representa una tabla en la base de datos. Cada instancia es una fila.

@Getter
@Setter
@NoArgsConstructor
@AllArgsConstructor
@Entity                                    // "Esta clase es una tabla"
@Table(name = "articulo_embalaje")         // nombre de la tabla en BD
public class ArticuloEmbalaje {

    @Id                                    // clave primaria
    @GeneratedValue(strategy = GenerationType.IDENTITY) // autoincrement
    @Column(name = "id_art")
    private Long idArt;

    @ManyToOne                             // muchos articulos → una categoria
    @JoinColumn(name = "id_cat", nullable = false)
    private CategoriaEmbalaje categoria;

    @Column(name = "nombre", nullable = false, length = 100)
    private String nombre;

    @Column(name = "descripcion", length = 255)
    private String descripcion;

    @Column(name = "precio_vta", nullable = false)
    private BigDecimal precioVta;

    @Column(name = "activo", nullable = false)
    private Boolean activo = true;
}
@ManyToOne: Muchos ArticuloEmbalaje pertenecen a una CategoriaEmbalaje. JPA crea automaticamente el JOIN con la FK id_cat.

Repository

Interfaz que hereda de JpaRepository. Spring genera toda la implementacion SQL automaticamente.

@Repository
public interface ArticuloEmbalajeRepository
    extends JpaRepository<ArticuloEmbalaje, Long> {

    // Spring genera el SQL solo por el nombre del metodo:
    // SELECT * FROM articulo_embalaje WHERE id_cat = ?
    List<ArticuloEmbalaje> findByCategoria_IdCat(Long idCat);
}

Service

Contiene la logica de negocio. Es donde vive el cerebro de tu aplicacion.

@Service
@RequiredArgsConstructor  // inyecta repos via constructor
public class ArticuloEmbalajeServiceImpl implements ArticuloEmbalajeService {

    private final ArticuloEmbalajeRepository repository;
    private final CategoriaEmbalajeRepository categoriaRepository;

    @Override
    public ArticuloEmbalajeResponse crear(ArticuloEmbalajeRequest request) {

        // 1. Buscar la categoria (validar que existe)
        CategoriaEmbalaje categoria = categoriaRepository.findById(request.getIdCat())
            .orElseThrow(() -> new RecursoNoEncontradoException("Categoria no encontrada"));

        // 2. Mapear DTO → Entity
        ArticuloEmbalaje articulo = new ArticuloEmbalaje();
        articulo.setCategoria(categoria);
        articulo.setNombre(request.getNombre());
        articulo.setDescripcion(request.getDescripcion());
        articulo.setPrecioVta(request.getPrecioVta());
        articulo.setActivo(true);

        // 3. Guardar y mapear Entity → Response DTO
        return toResponse(repository.save(articulo));
    }

    private ArticuloEmbalajeResponse toResponse(ArticuloEmbalaje a) {
        ArticuloEmbalajeResponse r = new ArticuloEmbalajeResponse();
        r.setIdArt(a.getIdArt());
        r.setIdCat(a.getCategoria().getIdCat());
        r.setNombreCategoria(a.getCategoria().getNombreCategoria());
        r.setNombre(a.getNombre());
        r.setDescripcion(a.getDescripcion());
        r.setPrecioVta(a.getPrecioVta());
        r.setActivo(a.getActivo());
        return r;
    }
}

DTOs (Request / Response)

DTOs separan la API publica del modelo interno:

// REQUEST: lo que el cliente ENVIA
@Data
public class ArticuloEmbalajeRequest {

    @NotNull(message = "La categoria es obligatoria")
    private Long idCat;

    @NotBlank(message = "El nombre es obligatorio")
    private String nombre;

    private String descripcion;   // opcional: sin @NotBlank

    @NotNull @Positive
    private BigDecimal precioVta;
}

// RESPONSE: lo que el servidor DEVUELVE
@Data
public class ArticuloEmbalajeResponse {
    private Long idArt;
    private Long idCat;
    private String nombreCategoria;  // dato extra que no esta en Request
    private String nombre;
    private String descripcion;
    private BigDecimal precioVta;
    private Boolean activo;
}
Por que no exponer la Entity directamente? Porque la entity tiene relaciones JPA (@ManyToOne) que causan referencias circulares al serializar a JSON. Ademas, el Response puede incluir campos calculados o de otras entidades que el Request no necesita.

Controller

Recibe las peticiones HTTP, valida, y delega al Service:

@RestController                        // respuestas en JSON automaticamente
@RequestMapping("/api/articulos")       // prefijo de URL
@RequiredArgsConstructor
public class ArticuloEmbalajeController {

    private final ArticuloEmbalajeService service;

    @PostMapping                           // POST /api/articulos
    public ResponseEntity<ArticuloEmbalajeResponse> crear(
            @Valid @RequestBody ArticuloEmbalajeRequest request) {
        return ResponseEntity.status(HttpStatus.CREATED)
            .body(service.crear(request));
    }

    @GetMapping                            // GET /api/articulos
    public ResponseEntity<List<ArticuloEmbalajeResponse>> listar() {
        return ResponseEntity.ok(service.listarTodos());
    }

    @GetMapping("/{id}")                    // GET /api/articulos/5
    public ResponseEntity<ArticuloEmbalajeResponse> obtenerPorId(
            @PathVariable Long id) {
        return ResponseEntity.ok(service.obtenerPorId(id));
    }
}
AnotacionHTTPUso tipico
@GetMappingGETLeer / listar
@PostMappingPOSTCrear
@PutMappingPUTActualizar completo
@PatchMappingPATCHActualizar parcial
@DeleteMappingDELETEEliminar

Manejo de excepciones global

@RestControllerAdvice captura excepciones de TODOS los controllers y devuelve JSON limpio:

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(RecursoNoEncontradoException.class)
    public ResponseEntity<Map<String, String>> handleNotFound(
            RecursoNoEncontradoException ex) {
        Map<String, String> error = new HashMap<>();
        error.put("error", ex.getMessage());
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, String>> handleValidation(
            MethodArgumentNotValidException ex) {
        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getFieldErrors()
            .forEach(e -> errors.put(e.getField(), e.getDefaultMessage()));
        return ResponseEntity.badRequest().body(errors);
    }
}

Eureka (Service Discovery)

Eureka es el registro central. Cada microservicio al arrancar le dice a Eureka: "Hola, soy ms_embalaje y estoy en la IP 172.18.0.4, puerto 8006".

Eureka Server (puerto 8761) ↑ registro ↑ registro ↑ registro | | | ms_embalaje ms_finanzas ms_paquetes :8006 :8090 :8005 ↑ ↑ ↑ | descubrimiento (lb://MS_EMBALAJE) | +------ Gateway (:8080) ------+--------+ ↑ Cliente

El Gateway no hardcodea IPs — le pregunta a Eureka "donde esta MS_EMBALAJE?" y Eureka le responde con la IP y puerto actuales.

Configuracion del Eureka Server

# eureka_server/src/main/resources/application.yml
spring:
  application:
    name: eureka_server

server:
  port: 8761

eureka:
  client:
    register-with-eureka: false   # el server no se registra a si mismo
    fetch-registry: false
  instance:
    hostname: ${EUREKA_HOSTNAME:eureka-server}

Configuracion del Eureka Client (en cada MS)

# En application.properties de cada microservicio:
eureka.client.service-url.defaultZone=${EUREKA_URI:http://localhost:8761/eureka/}
eureka.instance.prefer-ip-address=true

Gateway

El Gateway es el punto de entrada unico. El cliente solo conoce localhost:8080; el Gateway redirige al MS correcto segun la URL.

# gateway/src/main/resources/application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: ms-embalaje
          uri: lb://MS_EMBALAJE           # lb = load balanced via Eureka
          predicates:
            - Path=/api/articulos/**, /api/categorias/**, /api/stock/**

        - id: ms-finanzas
          uri: lb://MS_FINANZAS
          predicates:
            - Path=/api/cajas/**, /api/aperturas/**, /api/movimientos/**

        - id: ms-paquetes
          uri: lb://MS_PAQUETES
          predicates:
            - Path=/api/inventario/**
lb://MS_EMBALAJE significa: "busca en Eureka un servicio llamado MS_EMBALAJE y hazle load balancing". El nombre debe coincidir exactamente con el spring.application.name del microservicio (en mayusculas).

Comunicacion entre microservicios

Cuando un MS necesita datos de otro, usa WebClient (HTTP asincrono). Ejemplo: ms_embalaje llamando a ms_sucursales:

// Se inyecta un WebClient configurado con Eureka
@Bean
public WebClient webClient(WebClient.Builder builder) {
    return builder.build();
}

// Llamada a otro MS:
SucursalDTO sucursal = webClient.get()
    .uri("http://MS-SUCURSALES/api/v1/sucursales/{id}", idSucursal)
    .retrieve()
    .bodyToMono(SucursalDTO.class)
    .block();

Dockerfile

Nuestro Dockerfile usa multi-stage build — dos etapas para que la imagen final sea liviana:

# ETAPA 1: Compilar con Maven (imagen grande, ~800MB)
FROM maven:3.9-eclipse-temurin-25 AS build
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

# ETAPA 2: Solo el JAR + JRE (imagen chica, ~250MB)
FROM eclipse-temurin:25-jre
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
Por que multi-stage? La etapa de build tiene Maven + JDK (~800MB). La etapa final solo tiene JRE (~250MB). El COPY --from=build copia solo el JAR compilado, descartando todo lo demas.

Docker Compose

Orquesta multiples contenedores con un solo comando. Nuestro docker-compose.yml:

services:

  mariadb:                           # Base de datos compartida
    image: mariadb:11
    environment:
      MARIADB_ALLOW_EMPTY_ROOT_PASSWORD: "yes"
    ports: ["3306:3306"]
    volumes: [mariadb_data:/var/lib/mysql]  # datos persisten
    healthcheck: ...                        # Docker verifica que este lista

  eureka-server:                     # Service Discovery
    build: ./eureka_server
    ports: ["8761:8761"]
    environment:
      EUREKA_HOSTNAME: eureka-server
    healthcheck: ...

  gateway:                           # Punto de entrada
    build: ./gateway
    ports: ["8080:8080"]
    environment:
      EUREKA_URI: http://eureka-server:8761/eureka/
    depends_on:
      eureka-server: { condition: service_healthy }

  ms_embalaje:                       # Tu microservicio
    build: ./ms_embalaje
    ports: ["8006:8006"]
    environment:
      DB_HOST: mariadb                     # nombre del servicio = hostname
      EUREKA_URI: http://eureka-server:8761/eureka/
    depends_on:
      mariadb: { condition: service_healthy }
      eureka-server: { condition: service_healthy }

Flujo de arranque

1. mariadb arranca → healthcheck verifica conexion 2. eureka-server arranca → healthcheck verifica /eureka/apps 3. Cuando ambos estan healthy: → gateway, ms_embalaje, ms_finanzas, ms_paquetes arrancan en paralelo 4. Cada MS se registra en Eureka 5. Gateway descubre los MS via Eureka 6. Cliente accede a todo via gateway:8080

Flujo completo de trabajo

Para desarrollar localmente (sin Docker)

  1. Tener MariaDB corriendo en localhost:3306
  2. Levantar Eureka Server (puerto 8761)
  3. Levantar Gateway (puerto 8080)
  4. Levantar tu microservicio
  5. Probar via http://localhost:8080/api/...

Para levantar con Docker

cd A:\GitHub
docker compose up --build        # primera vez
docker compose up -d             # despues (en segundo plano)
# Abrir http://localhost:8761 → ver Eureka dashboard
# Abrir sivebo-test.html → probar los endpoints

Para crear un microservicio nuevo desde cero

  1. Ir a start.spring.io y seleccionar: Spring Web, Spring Data JPA, Validation, MariaDB Driver, Lombok, Eureka Discovery Client
  2. Descargar y descomprimir en A:\GitHub\ms_nombre\
  3. Crear la estructura de carpetas: 
    src/main/java/com/sivebo/ms_nombre/
├── controller/      ← Endpoints REST
├── service/         ← Interfaces de logica
│   └── impl/        ← Implementaciones
├── repository/      ← Acceso a BD
├── model/
│   ├── entity/      ← Entidades JPA
│   └── enums/       ← Enumeraciones
├── dto/
│   ├── request/     ← Lo que el cliente envia
│   └── response/    ← Lo que el servidor responde
├── exception/       ← Excepciones + handler global
└── config/          ← Configuraciones (WebClient, etc)
  4. Crear Entity → Repository → Service (interfaz + impl) → DTO → Controller → ExceptionHandler
  5. Configurar application.properties con Eureka y BD
  6. Agregar @EnableDiscoveryClient en la clase principal
  7. Crear Dockerfile
  8. Agregar al docker-compose.yml
  9. Agregar ruta en el Gateway

Preguntas tipicas de defensa

Arquitectura

"Por que microservicios y no un monolito?"

Porque SIVEBO tiene dominios claramente separados (embalaje, finanzas, paquetes, sucursales, etc.) que pueden evolucionar independientemente. Si el equipo de finanzas necesita un cambio, se modifica y despliega solo ms_finanzas sin afectar a los demas. Ademas, si un servicio falla, los otros siguen funcionando.

"Que pasa si Eureka se cae?"

Los microservicios que ya se registraron mantienen una cache local del registro. Pueden seguir comunicandose entre si por un tiempo. Pero nuevos servicios no podran registrarse hasta que Eureka vuelva. En produccion se usan multiples instancias de Eureka (cluster) para evitar esto.

"Por que un Gateway?"

El cliente solo necesita conocer una URL (localhost:8080). El Gateway se encarga del enrutamiento, y en el futuro podemos agregar autenticacion centralizada, rate limiting, CORS y logging sin tocar los microservicios.

"Que es lb:// en las rutas del Gateway?"

lb:// significa load balanced. El Gateway le pregunta a Eureka la IP del servicio y si hay multiples instancias, distribuye las peticiones entre ellas.

Spring Boot

"Que hace @SpringBootApplication?"

Es una meta-anotacion que combina: @Configuration (clase de configuracion), @EnableAutoConfiguration (configura todo automaticamente segun las dependencias), y @ComponentScan (escanea el package buscando @Service, @Controller, @Repository).

"Por que usas interfaz + implementacion en los Services?"

Principio de Inversion de Dependencias (SOLID). El Controller depende de la abstraccion (interfaz), no de la implementacion concreta. Esto facilita testing (puedes mockear) y permite cambiar la implementacion sin modificar los consumidores.

"Que es la inyeccion de dependencias?"

Spring crea los objetos (beans) y los conecta automaticamente. Cuando un Controller necesita un Service, no hace new ServiceImpl(). Spring detecta la dependencia y la inyecta via constructor (gracias a @RequiredArgsConstructor). Esto desacopla las clases entre si.

"Por que DTOs y no exponer la Entity directamente?"

Tres razones: (1) La Entity tiene relaciones JPA que pueden causar recursion infinita al serializar a JSON. (2) El Response puede incluir datos de multiples entidades. (3) El Request no necesita campos como ID o fechas que se generan automaticamente.

"Que hace @Valid?"

Activa la validacion automatica del DTO antes de que el metodo se ejecute. Si un campo tiene @NotNull y llega null, Spring lanza MethodArgumentNotValidException que nuestro GlobalExceptionHandler captura y devuelve como JSON con status 400.

"Que es JPA y que es Hibernate?"

JPA (Jakarta Persistence API) es la especificacion — define las anotaciones como @Entity, @Id, @ManyToOne. Hibernate es la implementacion que realmente ejecuta el SQL. Spring Boot usa Hibernate por defecto.

Docker

"Que ventaja tiene Docker sobre instalar todo en la maquina?"

Cualquier persona del equipo ejecuta docker compose up y tiene el mismo entorno exacto: misma version de Java, MariaDB, configuracion. No hay "en mi maquina funciona". Ademas, es identico al entorno de produccion.

"Que es un multi-stage build?"

Un Dockerfile con dos FROM. La primera etapa compila con Maven + JDK (imagen grande). La segunda etapa solo copia el JAR y usa JRE (imagen chica). La imagen final pesa ~250MB en vez de ~800MB.

"Por que depends_on con condition: service_healthy?"

Porque depends_on solo garantiza que el contenedor inicio, no que la aplicacion esta lista. Con service_healthy, Docker espera a que el healthcheck pase (Eureka responde, MariaDB acepta conexiones) antes de arrancar los servicios dependientes.

"Que es la variable ${DB_HOST:localhost}?"

Es una variable de entorno con valor por defecto. En Docker Compose inyectamos DB_HOST=mariadb (nombre del servicio = hostname en la red Docker). En desarrollo local, como no hay variable, usa localhost.
