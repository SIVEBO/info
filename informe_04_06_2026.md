# Informe de Errores — SIVEBO

## Contexto

SIVEBO es un sistema POS integral para la empresa RapidinBombin, construido con arquitectura de microservicios Spring Boot (14 servicios). El análisis compara el README (`info/README.md`) como especificación de referencia contra el código fuente, las configuraciones Maven y los modelos de datos implementados.

---

## ERRORES CRÍTICOS

### ERR-01 · Typo en `artifactId` de ms_finanzas
**Archivo:** `ms_finanzas/pom.xml` líneas 12 y 14  
**Código actual:**
```xml
<artifactId>ms_finanazas</artifactId>
<name>ms_finanazas</name>
```
**Problema:** El artefacto Maven se llama `ms_finanazas` (con "a" extra). El JAR generado tendrá ese nombre incorrecto, rompiendo scripts de despliegue y referencias cruzadas.  
**Corrección:** Cambiar a `ms_finanzas`.

---

### ERR-02 · Configuración de seguridad completamente permisiva
**Archivo:** `ms_auth/src/main/java/com/sivebo/ms_auth/config/SecurityConfig.java` líneas 26–27  
**Código actual:**
```java
.requestMatchers("/auth/**").permitAll()
.anyRequest().permitAll()  // ← Toda petición es pública
```
**Problema:** La línea `anyRequest().permitAll()` anula cualquier protección de roles. Cualquier endpoint del sistema es accesible sin autenticación, incluyendo operaciones de administración, creación de usuarios y cierre de caja.  
**Corrección:** Cambiar a `.anyRequest().authenticated()` y configurar los roles correspondientes.

---

### ERR-03 · Tipo de retorno incorrecto en consulta nativa
**Archivo:** `ms_guias_despacho/src/main/java/com/sivebo/ms_guias_despacho/repository/EstadoMaestroRepository.java` línea 12  
**Código actual:**
```java
@NativeQuery("SELECT tipo_estado FROM estado_maestro WHERE tipo_estado = ?1")
public Long GetIdByTipoEstado(String tipoEstado);
```
**Problemas múltiples:**
1. La consulta selecciona `tipo_estado` (un ENUM `TipoEstado`), pero el tipo de retorno declarado es `Long`. Esto producirá un `ClassCastException` en tiempo de ejecución.
2. Si la intención es obtener el ID del estado, la consulta debe ser `SELECT id FROM estado_maestro WHERE tipo_estado = ?1`.
3. El nombre del método (`GetIdByTipoEstado`) contradice lo que realmente retorna la consulta actual.

---

### ERR-04 · Dependencias de test inexistentes en Maven
**Archivos:** Múltiples `pom.xml` — confirmado en `ms_finanzas/pom.xml` líneas 62 y 67  
**Código actual:**
```xml
<artifactId>spring-boot-starter-data-jpa-test</artifactId>
<artifactId>spring-boot-starter-validation-test</artifactId>
```
**Problema:** Estos artefactos no existen en Maven Central. El build fallará con `Could not resolve dependencies`. El starter correcto para tests es `spring-boot-starter-test`.  
**Corrección:**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
</dependency>
```

## ERRORES DE ALTA PRIORIDAD

### ERR-05 · Modelo `GuiaDespacho` diverge del esquema especificado
**Archivo:** `ms_guias_despacho/src/main/java/com/sivebo/ms_guias_despacho/model/GuiaDespacho.java`  
**Según README (sección 6, MS-04):** `GUIA_DESPACHO` debe tener `id_guia`, `codigo_tracking`, `id_admision`, `fecha_creacion`.  
**Implementación actual:**
```java
Long id;                      // PK llamada "id", no "id_guia"
String codigoTracking;        // ✓
Long idAdmision;              // ✓
Long idEstadoMaestro;         // EXTRA — no está en la especificación
// fecha_creacion             FALTA — ausente del modelo
```
**Problemas:**
1. `id_estado_maestro` no pertenece a `GUIA_DESPACHO` según el diseño; el estado se gestiona en `HISTORIAL_LOGISTICO`.
2. El campo `fecha_creacion` está ausente del modelo Java y del constructor Lombok.
3. La PK se llama `id` en lugar de `id_guia` (sin `@Column(name="id_guia")`), inconsistente con la nomenclatura del proyecto.

---

### ERR-06 · JOIN incorrecto en `GuiaDespachoRepository`
**Archivo:** `ms_guias_despacho/src/main/java/com/sivebo/ms_guias_despacho/repository/GuiaDespachoRepository.java` líneas 17–21  
**Código actual:**
```java
@NativeQuery(
    "SELECT * FROM guia_despacho " +
    "JOIN estado_maestro ON guia_despacho.id_estado_maestro = estado_maestro.id " +
    "WHERE estado_maestro.tipo_estado = ?1")
List<GuiaDespacho> findByEstadoMaestro(String estadoMaestro);
```
**Problemas:**
1. El JOIN referencia `guia_despacho.id_estado_maestro`, columna que no existe en el esquema del README.
2. El `SELECT *` en un JOIN devuelve columnas de ambas tablas, pero el resultado se mapea a `List<GuiaDespacho>`, causando error de mapeo al encontrar columnas de `estado_maestro`.
3. Según el diseño, para filtrar guías por estado se debe consultar `HISTORIAL_LOGISTICO` o la vista `V_ESTADO_ACTUAL_GUIA`, no hacer JOIN directo en `GUIA_DESPACHO`.

---

### ERR-07 · Nombre de base de datos incorrecto en ms_detalle_venta
**Archivo:** `ms_detalle_venta/src/main/resources/application.properties` línea 4  
**Código actual:**
```properties
spring.datasource.url=jdbc:mariadb://localhost:3306/db_ms_detalle_ventas?...
```
**Problema:** La base de datos se llama `db_ms_detalle_ventas` (plural) mientras el microservicio se llama `ms_detalle_venta` (singular). Si el script DDL crea la BD como `db_ms_detalle_venta`, el servicio no podrá conectarse.

---

### ERR-08 · Typo en URL de endpoint del portal
**Archivo:** `ms_portal/src/main/java/com/sivebo/ms_portal/controller/FeedbackController.java` línea 27  
**Código actual:**
```java
@RequestMapping("api/v1/feebacks")
```
**Problema:** El endpoint es `feebacks` (falta la "d"). Debería ser `feedbacks`. Cualquier cliente que llame a `/api/v1/feedbacks` recibirá un 404.

---

### ERR-09 · Conversión de tipo sin manejo de excepciones
**Archivo:** `ms_portal/src/main/java/com/sivebo/ms_portal/controller/FeedbackController.java` línea 63  
**Código actual:**
```java
return ResponseEntity.ok(
    feedbackService.getbyIdGuiaTracking(Long.valueOf(idGuiaTracking)));
```
**Problema:** Si `idGuiaTracking` no es un número válido (ej. `"abc"` o `""`), `Long.valueOf()` lanzará `NumberFormatException` no capturada, devolviendo un HTTP 500 al cliente. Debe validarse con un bloque try-catch o un `@ExceptionHandler`.

## ERRORES DE PRIORIDAD MEDIA

### ERR-10 · Credenciales de base de datos hardcodeadas
**Archivos:** Todos los `application.properties` (14 servicios)  
**Código actual:**
```properties
spring.datasource.username=root
spring.datasource.password=
```
**Problema:** Credenciales de root expuestas en texto plano en archivos de configuración versionados en Git. Cualquier persona con acceso al repositorio tiene las credenciales de producción.  
**Corrección:** Usar variables de entorno o Spring Profiles (`application-prod.properties`).

---

### ERR-12 · Inconsistencia en la propiedad de dialecto Hibernate
**Archivos:** `application.properties` en varios microservicios  
**Dos formatos distintos en uso:**
```properties
# Formato A (ms_comunas, ms_regiones, ms_sucursales):
spring.jpa.database-platform=org.hibernate.dialect.MariaDBDialect

# Formato B (ms_auth, ms_clientes, ms_admision):
spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.MariaDBDialect
```
**Problema:** Solo el Formato A es correcto para Spring Boot 3+. El Formato B es una propiedad de Hibernate legacy que puede no aplicarse. Además, varios servicios (`ms_finanzas`, `ms_embalaje`, `ms_paquetes`, `ms_guias_despacho`) no tienen el dialecto definido.

---

### ERR-13 · Concatenación de URI sin codificación en WebClientUtil
**Archivos:** `ms_historial_logistico/src/.../utils/WebClientUtil.java` línea 34, y equivalentes en `ms_ventas` y `ms_portal`  
**Código actual:**
```java
.uri("/api/v1/" + name_service + "/search?" + query + "=" + value, value)
```
**Problema:** Los parámetros se concatenan directamente sin URL encoding. Si `value` contiene caracteres especiales (`&`, `=`, espacios), la URI se corrompe. Debe usarse el builder de URI de WebClient: `.uri(uriBuilder -> uriBuilder.path(...).queryParam(...).build())`.

---

### ERR-14 · `JpaRepository` tipado como `Object` en lugar de `Long`
**Archivos:**  
- `ms_guias_despacho/.../EstadoMaestroRepository.java`: `JpaRepository<EstadoMaestro, Object>`  
- `ms_guias_despacho/.../GuiaDespachoRepository.java`: `JpaRepository<GuiaDespacho, Object>`  
**Problema:** El tipo genérico del ID debería ser `Long` (el tipo real de la PK). Usar `Object` anula los beneficios del tipado estático y puede causar problemas con los métodos `findById(Long)` y `deleteById(Long)`.

## BRECHAS DE DISEÑO (no son bugs, pero bloquean la especificación)

### GAP-01 · API Gateway / Eureka no implementado
**README sección 3:** "API Gateway (Eureka)" es **obligatorio**.  
**Estado:** Ningún `pom.xml` incluye dependencias de Spring Cloud Netflix (`spring-cloud-starter-netflix-eureka-server` o `eureka-client`). No existe microservicio de gateway en el proyecto.

### GAP-02 · JWT solo existe en ms_auth, sin validación en los demás servicios
**README sección 5:** "Validación de accesos según roles definidos" para todos los servicios.  
**Estado:** Solo `ms_auth` tiene `spring-boot-starter-security`. Los otros 13 microservicios no validan el token JWT en sus endpoints.

## Verificación

Para confirmar los errores antes de corregir:
1. `mvn clean package` en `ms_finanzas/` → fallará por typo en artifactId y dependencias de test inexistentes.
2. `mvn clean package` en `ms_guias_despacho/` → fallará al resolver la referencia a `id_estado_maestro` en el JOIN.
3. Llamar `GET http://localhost:{puerto}/api/v1/feebacks` en ms_portal → 404 (typo en URL).
4. Llamar `GET http://localhost:{puerto}/api/v1/feedbacks?id=abc` → HTTP 500 (NumberFormatException).
5. Acceder a cualquier endpoint protegido sin token → HTTP 200 (seguridad permisiva confirmada).
